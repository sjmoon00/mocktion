# 스코프 트레이드오프 기록 (Tradeoff Log)

> 이 문서는 "무엇을 의도적으로 하지 않았고, 왜 그게 지금 단계에 맞는 선택인가"를 기록한다.
> 무엇을 선택했는지의 근거는 [DECISIONS.md](DECISIONS.md) 참고. 여기 적힌 항목 중 사용자에게 영향을 주는 것은 README의 "알려진 한계" 섹션에도 그대로 옮긴다.

각 항목: **포기한 것 / 얻은 것 / 왜 지금은 이게 맞는가**

---

## T-001: Request 섹션(Path Parameter / Request Body / Header) 파싱 안 함

- **포기한 것**: Path Parameter, Request Body, Request Header 테이블의 구조화된 파싱.
- **얻은 것**: 페이지마다 컬럼 수(3열/4열/5열)와 섹션 이름(`Request Body` vs `Request Fields`)이 달라도 파서가 영향받지 않는 단순함.
- **왜 맞는가**: 목서버는 요청 내용을 검증하지 않는다. Path Parameter는 DB의 URI 프로퍼티에서 `{변수}` 패턴을 정규식으로 뽑는 것으로 충분하다. 애초에 필요 없는 데이터의 파싱 불일치를 신경 쓸 이유가 없다.

---

## T-002: 쿼리 파라미터 기반 에러 응답 트리거(`?error=404`) 미구현

- **포기한 것**: 프론트가 특정 에러 케이스를 의도적으로 재현해보는 기능.
- **얻은 것**: Phase 1의 스코프를 "성공 응답 자동화"로 좁혀 먼저 동작하는 것을 빠르게 만듦.
- **왜 맞는가**: 에러 케이스는 콘솔에 목록으로 출력해 프론트가 참고는 할 수 있게 하고, 실제 트리거는 Phase 2 이후로 미룬다. 배포를 마지막으로 미루지 않는다는 원칙(spec 9절)과 맞물려 "된다"를 먼저 만들고 "완전하다"는 나중.

---

## T-003: 여러 상태값 동시 필터(`--status 완료,검토`) 미지원

- **포기한 것**: 상태값 다중 선택.
- **얻은 것**: `--status` 옵션 구현이 단순한 equals 비교로 끝남.
- **왜 맞는가**: 우선순위가 낮은 편의 기능. 단일 상태값 필터로도 "이번 스프린트에 완료된 것만 실행"이라는 핵심 시나리오는 충분히 커버된다.

## T-005: 요청 예시 JSON이 응답으로 오분류될 수 있는 설계상 한계

- **포기한 것**: 100% 정확한 요청/응답 분류. heading/language 태그를 신뢰하지 않는 대신, 요청 본문 JSON이 페이지에서 별도의 독립된 `{...}` 코드 블록으로 분리되어 있는 특이 케이스는 응답으로 오분류될 수 있다.
- **얻은 것**: [D-004](DECISIONS.md#d-004-heading-텍스트code-블록-language-태그를-신뢰하지-않고-내용-패턴으로-분류)에서 확인했듯, heading이 없거나 오타가 있는 실제 페이지에서도 안정적으로 동작하는 견고함.
- **왜 맞는가**: 실 데이터 검증(4개 페이지) 기준으로 이 오분류가 실제로 발생한 사례는 없었다. 발생 가능성이 낮은 엣지 케이스를 잡으려고 복잡한 휴리스틱을 추가하기보다, README에 알려진 한계로 명시하는 쪽을 택했다.

---

## T-006: 응답 후보 code 블록이 여러 개일 때 첫 번째만 사용

- **포기한 것**: 여러 응답 후보 중 "진짜" 응답을 정교하게 골라내는 로직.
- **얻은 것**: 규칙이 단순해 예측 가능하고, 콘솔 경고로 사용자가 직접 확인할 수 있음.
- **왜 맞는가**: 실제 4개 검증 예시에서 이 케이스가 나온 적이 없다. 발생하지 않는 것으로 확인된 케이스에 미리 정교한 규칙을 만드는 건 과설계.

---

## T-007: 변경 감지 자동 리로드 / 웹 UI 대시보드 / 클라우드 배포 제외

- **포기한 것**: 파일 변경 감지 후 자동 재시작, 브라우저 대시보드, 클라우드에 올려서 공유하는 기능.
- **얻은 것**: "CLI 한 줄 → 로컬 목서버 즉시 실행"이라는 핵심 가치에 집중.
- **왜 맞는가**: 이 도구의 문제 정의는 "Notion 명세를 로컬에서 즉시 목서버로 쓰고 싶다"이지 "목서버 플랫폼을 만들고 싶다"가 아니다. 스코프를 넓히면 핵심 가치가 희석된다.

---

## T-008: Request Body 검증 없음

- **포기한 것**: 클라이언트가 보낸 요청 바디가 명세와 일치하는지 검사.
- **얻은 것**: 서버 로직이 "URI+Method 매칭 → 정해진 응답 반환"으로 끝나는 단순함.
- **왜 맞는가**: 목서버의 목적은 프론트가 응답 형태를 미리 보고 개발하는 것이지, 백엔드 계약 테스트가 아니다. 검증이 필요하면 별도 계약 테스트 도구의 역할이다.
